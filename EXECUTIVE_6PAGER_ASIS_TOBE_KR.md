대상: 경영진, 사업부 리더, 운영/기획 책임자

## 1. Executive Summary

우리 조직은 HWPX 문서 처리 역량을 이미 확보하고 있다. 현재 자산은 세 축으로 구성된다. 첫째, 문서 구조를 다루는 C++ OWPML 모델(`hwpx-owpml-model-src`). 둘째, 실제 렌더링과 읽기를 담당하는 Java 리더(`hwpx-reader-src`). 셋째, AI 기반 편집과 배포를 담당하는 웹 앱(`hwpx-report-automation/web`)이다. 즉, 기술 자산은 충분히 축적되었지만, 사용자 경험과 운영 체계가 하나의 제품 경험으로 완전히 통합되지는 않았다.

현재(As-Is)는 “개발자 중심 자동화” 단계다. 파일 단위 스크립트와 모듈별 운영이 가능하고, 문서 손상 방지(무결성 검사)와 부분 수정(텍스트/표 patch)까지 구현되어 있다. 다만 임원 관점에서 중요한 것은 기능 존재 자체가 아니라, 비개발자도 반복적으로 쓸 수 있는 “업무 시스템”인지 여부다. 이 지점에서 아직 생산성의 상한이 존재한다.

목표(To-Be)는 “비개발자도 즉시 사용할 수 있는 문서 생산 플랫폼”이다. 단일 WYSIWYG 편집, 섹션 단위 AI 일괄 개선, 변경 이력/감사 추적, 대용량 문서 성능 최적화, 외부 결재/협업 시스템 연계를 포함한 운영형 플랫폼으로 전환한다. 핵심은 기술 고도화가 아니라, 문서 작성 시간을 단축하고 품질 편차를 줄여 조직 전체의 실행 속도를 높이는 것이다.

이번 제안의 결론은 명확하다. 기존 자산을 폐기하지 않고, 현재 구조를 “제품화”하면 된다. 이미 구현된 기반(양방향 변환, 무결성 검증, AI 제안 API, 테스트 체계)을 활용해 2개 분기 내에 실사용 수준으로 전환 가능하다.

## 2. 배경과 문제 정의 (Why Now)

문서 중심 조직에서는 보고서/대외 문서/사업 제안서의 생산성과 일관성이 사업 속도를 좌우한다. 그러나 전통적인 HWPX 편집 방식은 다음의 구조적 한계를 갖는다.

첫째, 편집 방식이 도구별로 분절되어 있다. 원문 탐색, 수정, 검증, 배포가 하나의 흐름으로 연결되지 않아, 사람이 여러 단계에서 수작업으로 확인한다. 이 과정에서 납기 지연과 품질 편차가 반복된다.

둘째, AI 활용이 “초안 생성”에 머물면 생산성 효과가 제한된다. 실제 현업은 문서 전체를 다시 편집하고, 형식 오류를 잡고, 최종 파일이 열리는지 검증해야 한다. 즉, 생성보다 중요한 것은 “안전한 반영”과 “운영 가능성”이다.

셋째, 임원 보고 문서는 기술적 정확성뿐 아니라 문장 일관성, 톤, 결론 구조가 중요하다. 현재 구조는 기능적으로 충분하지만, 비개발자가 즉시 활용하기에는 운영 가이드/권한 정책/품질 지표의 제품화가 더 필요하다.

지금 전환이 필요한 이유는, 기술 리스크가 아니라 기회비용 때문이다. 이미 구현된 자산 위에서 제품화만 진행하면, 단기간에 문서 생산 리드타임 단축과 품질 표준화를 동시에 달성할 수 있다.

## 3. 현재 상태(As-Is) 진단

### 3.1 현재 프로젝트 구조(확인 기준)

현재 저장소 구조는 다음과 같이 역할이 분명하다.

- `hwpx-owpml-model-src`: OWPML 기반 문서 모델(C++), 저수준 구조 처리 자산
- `hwpx-reader-src`: Java 기반 HWPX 읽기/렌더링 자산
- `hwpx-report-automation`: 자동 편집 파이프라인과 웹 제품화 레이어
  - `scripts/`: Python 기반 템플릿 치환/리포트 생성 자동화
  - `web/`: Next.js + TipTap + Zustand 기반 인터랙티브 편집 앱

이 구조는 “엔진(모델) + 리더(렌더) + 제품(UI)”의 형태로 확장 가능성이 높다.

### 3.2 As-Is 강점

현재 웹 앱은 실사용 핵심 요소를 이미 갖췄다.

- WYSIWYG 편집(문서형 UI), 표 편집, 슬래시 커맨드
- AI 단건 제안(`/api/suggest`) 및 일괄 제안(`/api/suggest-batch`)
- HWPX 무결성 검증(필수 엔트리/XML 파싱)
- XML 전체 재작성 대신 부분 patch(손상 리스크 완화)
- 변경 이력, Outline, before/after diff 패널
- 테스트 체계 가동: 최근 실행 기준 8개 파일, 18개 테스트 통과

즉, “기술 검증 단계”는 이미 통과했고 “운영 제품 단계”로 넘어갈 수 있는 상태다.

### 3.3 As-Is 한계

경영 관점에서 남은 과제는 다음이다.

- 사용자 경험 분절: 고급 기능이 늘었지만 업무 템플릿/역할별 UX가 아직 약함
- 운영 정책 부재: 프롬프트/모델/비용/권한/감사 로그의 표준 운영정책이 미완성
- 구조 반영 제약: 신규 표(원본 tableId 없는 객체)의 완전 반영은 제한적
- 보안/거버넌스 리스크: 키 관리/비밀정보 관리 체계 강화 필요
- 연계 미완성: Java 렌더 API, 웹 편집, 배치 자동화가 완전한 단일 운영 플로우로 통합되지는 않음

결론적으로, 현재는 “기술적으로 가능”하지만 “조직적으로 확산 가능한 상태”까지는 아직 1~2단계가 남아 있다.

## 4. 목표 상태(To-Be): 비개발자 중심 문서 생산 플랫폼

To-Be는 기능 추가보다 운영 전환에 초점을 둔다. 핵심 목표는 다음 세 가지다.

첫째, 누구나 사용하는 단일 편집 경험이다. 사용자는 파일 업로드 후 문서를 그대로 보며 수정하고, AI 제안을 섹션 단위로 적용하고, 검증된 파일을 즉시 내려받는다. “개발자 도움 없이” 반복 가능한 업무가 되어야 한다.

둘째, 품질과 속도의 동시 달성이다. 자동화는 빨라야 하지만, 손상 없는 결과가 전제다. 따라서 부분 patch, 무결성 검사, diff 검토, 이력 추적을 기본값으로 유지한다. 초안 생성보다 최종 산출물의 신뢰성을 우선한다.

셋째, 경영 가능한 운영체계다. 모델/프롬프트/권한/비용/감사 기준을 운영정책으로 관리해, 담당자 변경이나 조직 확장 시에도 품질이 유지되도록 한다.

### 4.1 As-Is → To-Be 전환표

| 영역 | As-Is | To-Be |
|---|---|---|
| 사용자 경험 | 기능 중심(전문 사용자 친화) | 역할 중심(기획/운영/임원 보좌도 즉시 사용) |
| 편집 안정성 | 텍스트/표 부분 patch + 무결성 검사 | 정책 기반 자동검증 + 실패 복구 + 승인 워크플로우 |
| AI 활용 | 단건/일괄 제안 제공 | 문서 유형별 가이드 내장 + 품질 점수 + 승인 루프 |
| 운영 통제 | 환경변수 중심 개별 관리 | 키/모델/비용/권한 중앙 통제 |
| 확장성 | 저장소 단위 모듈 분리 | 단일 제품 포털 + 부서별 템플릿 카탈로그 |
| 경영 리포팅 | 개발 로그 중심 | KPI 대시보드(시간절감, 재작업률, 오류율) |

## 5. 실행 계획 (2개 분기 로드맵)

### Q1: 제품 안정화와 표준화

목표는 “전사 파일 1차 적용 가능 상태”다.

- 문서 유형별 템플릿/프롬프트 표준화(보고서, 제안서, 공문)
- 신규 표 생성/삽입 반영 범위 확대 및 예외 처리 정책 정리
- 운영 콘솔 초안: 모델 선택, 비용 한도, 사용자 권한, 감사 로그
- UAT(비개발 사용자) 기반 UX 개선: 용어 단순화, 버튼 시나리오 정리
- 품질 게이트 자동화: 테스트 + 샘플 문서 회귀 + 무결성 리포트

성공 기준은 “비개발 사용자 10명 이상이 별도 지원 없이 문서 작성→검토→배포를 완료”하는 것이다.

### Q2: 운영 고도화와 확산

목표는 “부서 단위 상시 운영”이다.

- 부서별 템플릿 카탈로그와 문체 가이드 적용
- 협업 워크플로우(검토자 승인, 버전 비교, 이력 조회) 강화
- 대용량 문서 처리 최적화(배치 처리, 백그라운드 큐, 진행률 표시)
- Java 렌더링 연계 기능을 운영 플로우에 통합(보기 신뢰성 강화)
- KPI 대시보드 운영(리드타임, 재작업률, 오류율, AI 적용률)

성공 기준은 “핵심 문서군의 평균 작성시간 30% 단축, 재작업률 20% 감소”다.

## 6. 리스크, 대응, 경영진 의사결정 요청

### 6.1 주요 리스크와 대응

리스크 1은 보안/컴플라이언스다. API 키, 민감 문서, 외부 모델 호출 정책이 동시에 관리되어야 한다. 대응은 키 로테이션, 비밀관리 분리, 문서 등급별 처리 정책, 감사 로그 의무화다.

리스크 2는 품질 일관성이다. AI 결과가 문서 유형별 기대 수준을 벗어날 수 있다. 대응은 템플릿별 품질 룰셋, 인간 승인 단계, 자동 diff 점검이다.

리스크 3은 확산 시 운영 부담이다. 사용자 수가 늘면 지원 요청도 증가한다. 대응은 셀프 가이드, 표준 시나리오, 운영 대시보드, 장애 대응 런북 정착이다.

### 6.2 경영진 의사결정 요청사항

1. 본 과제를 “PoC 종료”가 아닌 “제품화 전환 과제”로 공식 승인
2. Q1~Q2 동안 전담 인력(제품 1, 프론트 1, 백엔드/플랫폼 1, QA/운영 1) 배정
3. 보안/법무/운영 부서와의 공통 정책(문서 등급, AI 사용범위, 로그보존) 확정

이 결정이 내려지면, 현재 자산을 유지한 채 비용 효율적으로 문서 자동화 체계를 전사 표준으로 전환할 수 있다.
